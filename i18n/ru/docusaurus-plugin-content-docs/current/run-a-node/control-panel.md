# Панель управления dig-node

В DIG Chrome extension встроена **панель управления** для вашего локального dig-node. С её помощью вы можете видеть текущий статус узла, решать, сколько дискового пространства зарезервировать под кэшированный контент, а после однократного сопряжения — управлять вышестоящим узлом (upstream), хранилищами, которые узел раздаёт, его синхронизацией и пирами. Для повседневного использования командная строка не требуется.

Панель управления — это встроенный в расширение аналог экрана управления узлом в DIG Browser: она общается с узлом, работающим на вашей собственной машине, поэтому всё остаётся локальным.

## Открыть панель

1. Откройте расширение.
2. Перейдите на вкладку **Network** (Сеть) и выберите **Control** (Управление). (В компактном всплывающем окне показана только сводка; чтобы увидеть каждый раздел на весь экран, используйте **Открыть панель управления**.)

Панель автоматически определяет узел:

- **Узел запущен** → отображается экран управления.
- **Узел не найден** → отображается короткая страница с инструкцией по установке узла. Просмотр контента по-прежнему работает — чтение контента переключается на публичную сеть; узел нужен только для экрана управления, описанного ниже.

## Статус в реальном времени

Вверху индикатор в реальном времени показывает, находится ли ваш узел в состоянии **Подключён**, **Подключение** или **Отключён**, вместе с его адресом и версией. Индикатор обновляется автоматически — запустите или остановите узел, и индикатор изменится в течение нескольких секунд, без необходимости заново открывать панель или обновлять страницу.

## Резервирование дискового пространства под кэшированный контент (кэш и LRU)

Ваш узел ведёт локальный кэш уже полученного контента, поэтому повторные обращения выполняются мгновенно, а вы помогаете раздавать этот контент. У кэша есть **зарезервированный размер** — предел того, сколько диска он может занять. Когда кэш превышает этот предел, узел автоматически удаляет в первую очередь **наименее недавно использованные** элементы (политика «LRU»), поэтому зарезервированное пространство никогда не превышается, а реально используемый вами контент остаётся в кэше.

Этот раздел доступен сразу — сопряжение для него не требуется.

**Посмотреть, сколько занято.** Индикатор показывает занятое пространство относительно зарезервированного предела, а также несколько показателей в реальном времени: сколько элементов кэшировано, их общий размер, сколько было вытеснено с момента запуска узла, а также число попаданий и промахов кэша.

**Задать зарезервированный предел.** Введите новый размер и примените его. Минимум — **64 MiB**; меньшее значение поднимается до этого порога. Если снизить предел ниже текущего использования, это запустит вытеснение самых старых элементов, пока использование не впишется в новый предел.

**Просмотр и удаление элементов кэша.** Список кэша показывает каждый элемент с его размером, временем последнего использования и **порядком вытеснения** (позиция `0` — элемент, который будет удалён следующим). Вы можете:

- **Вытеснить один элемент** — удалить один кэшированный элемент прямо сейчас.
- **Очистить всё** — полностью очистить кэш.

Удаление элементов освобождает только локальный диск; всё, к чему вы обратитесь снова, будет просто заново загружено.

:::tip
Выделите под кэш как можно больше места на машине, с которой вы часто просматриваете контент, — чем больше резерв, тем меньше повторных загрузок и тем больше контента раздаётся локально. На машине с ограниченным местом задайте меньший резерв; LRU сохранит самые полезные элементы и отбросит остальные.
:::

## Управление узлом (требуется сопряжение)

Остальные разделы изменяют конфигурацию узла, поэтому требуют вашего явного разрешения. Поскольку расширение работает в песочнице браузера, оно не может напрямую читать локальный файл разрешений узла — вместо этого вы один раз выполняете **сопряжение**. Сопряжение выдаёт расширению собственный, ограниченный по правам и отзываемый учётные данные; оно никогда не раскрывает мастер-ключ узла, и подтвердить его можно только с компьютера, на котором работает узел.

### Сопряжение расширения с вашим узлом

1. В панели управления выберите **Сопряжение**. Расширение покажет **6-значный код** и идентификатор сопряжения.
2. На компьютере, где работает узел, в терминале выполните `dig-node pair`, чтобы увидеть список ожидающих запросов (или сразу выполните `dig-node pair approve <pairing-id>`).
3. Убедитесь, что код, показанный в терминале, **совпадает** с кодом в расширении, и подтвердите. Это совпадение — ваша защита: оно гарантирует, что вы подтверждаете именно *это* расширение и ничего другого.
4. Панель управления автоматически переключится в состояние сопряжения. Учётные данные хранятся только в расширении.

Код сопряжения **истекает через несколько минут**; если ваш код истёк до того, как вы его подтвердили, выберите **Сопряжение** ещё раз, чтобы получить новый.

Чтобы прекратить использовать эти учётные данные, выберите в панели **Отменить сопряжение** (это просто забывает их локально). Чтобы отозвать их на самом узле, выполните на этом компьютере `dig-node pair revoke <token-id>` — при следующем действии панель вернётся в несопряжённое состояние.

:::note
Сопряжение нужно только для разделов управления ниже. Статус в реальном времени и элементы управления кэшем/LRU выше работают без него.
:::

### Вышестоящий узел

Посмотрите, с какого вышестоящего узла (upstream) ваш узел получает контент, и задайте другой. Изменённый вышестоящий узел вступит в силу при следующем запуске узла.

### Размещённые хранилища

Смотрите хранилища, которые держит и закрепляет ваш узел, закрепляйте новое хранилище (чтобы узел хранил и раздавал его), открепляйте существующее и проверяйте статус любого хранилища. Закрепление конкретной версии заранее загружает её, чтобы она была готова к раздаче.

### Синхронизация

Смотрите, доступна ли аутентифицированная синхронизация всего хранилища целиком, и запускайте синхронизацию для конкретной версии, чтобы узел получил и закэшировал её.

### Пиры

Смотрите статус пиринговой сети вашего узла — его соединение с релеем для доступности из-за домашнего маршрутизатора и пиры, с которыми он соединён.
